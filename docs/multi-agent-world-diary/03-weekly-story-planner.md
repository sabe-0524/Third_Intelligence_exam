# 03. Story Planner 設計

## 1. 責務

Story Planner は、7 日間全体の物語骨格を先に固定するモジュールである。自由生成の出力を面白くする役ではなく、後段が破ってはいけない制約を定義する役とみなす。

Story Planner は以下を `MUST` 出力する。

- 週テーマ
- 中心誤解
- 日ごとの物語役割
- A/B/C の週次アーク
- 各日に必要なズレ、補強、反転の条件
- 最終日まで秘匿すべき情報

## 2. 入力契約

最低入力は以下とする。

```json
{
  "premise_id": "library-lumen-misunderstanding-v1",
  "weekly_theme": "誤解",
  "fixed_characters": ["A", "B", "C"],
  "seed": 42
}
```

## 3. 出力契約

Story Planner の出力は、少なくとも以下の情報を持つ。

```json
{
  "week_run_id": "week-20260412-001",
  "daily_story_roles": [
    {
      "day_index": 1,
      "story_role": "interest",
      "must_reveal": ["Aの変化"],
      "must_hide": ["Aの本心", "Bの誤解"],
      "required_pressure": "軽い違和感"
    }
  ],
  "character_arcs": {
    "A": "避けているように見えつつ、実際は話したい側",
    "B": "距離を置かれていると誤解している側",
    "C": "解釈を崩す補助人物"
  }
}
```

## 4. 日別契約

各日は以下を満たさなければならない。

### Day 1

- `MUST`: A の変化が観測される。
- `MUST`: A と B が同じ空間にいたが接触しなかったように見える。
- `MUST NOT`: 明確な真相を示す。

### Day 2

- `MUST`: 主人公が `AはBを避けている` という仮説を立てる。
- `MUST`: 最初の介入が起こる。

### Day 3

- `MUST`: Day 2 の仮説を補強する観測が増える。
- `MUST`: 介入が意図からズレる。

### Day 4

- `MUST`: B 側の行動により仮説に裂け目が入る。
- `MUST`: C もしくは外部イベントが再解釈のきっかけを作る。

### Day 5

- `MUST`: A の行動が「回避」より「迷い」に見え始める。
- `MUST`: 介入の影響が B へ届く。

### Day 6

- `MUST`: 主人公が過去観測の再解釈を行う。
- `MUST`: 自己理解が更新される。

### Day 7

- `MUST`: 真相を全開示しない。
- `MUST`: Day 1 時点との見方の差が明示される。

## 5. A/B/C アーク契約

### A

- `MUST`: 可視行動だけを見ると B を避けているように見える。
- `MUST`: hidden truth では B に接触したい動機を持つ。
- `SHOULD`: 進路不安が行動選択に影響する。

### B

- `MUST`: A の変化に気づいている。
- `MUST`: A に距離を置かれているという誤解を持つ。
- `SHOULD`: Day 4 以降、迷いが可視行動に現れる。

### C

- `MUST`: 真相の説明役ではなく、解釈を崩す触媒として機能する。
- `MUST NOT`: hidden truth を直接言語化して暴露する。

## 6. Planner の禁止事項

- 7 日の役割を圧縮または欠落させることは `MUST NOT`。
- 中心誤解を途中で別テーマへ差し替えることは `MUST NOT`。
- 最終日に説明しすぎて余韻を壊すことは `MUST NOT`。

## 7. 検証観点

Story Planner の出力は、少なくとも以下で検証されなければならない。

- 全 7 日が埋まっているか。
- 日ごとの役割が重複しすぎていないか。
- `must_hide` に含めた情報が早期フェーズへ漏れていないか。
- A/B/C のアークが中心誤解と矛盾していないか。
