# 01. システム要件

## 1. 目的

本システムは、図書館内の返却棚・検索端末・推薦表示・案内サイネージを統合管理する観測 AI `Lumen` を主人公として、7日間の観測と小さな介入の結果を日記として出力する。

本システムが `MUST` 満たすべき価値は以下である。

- 主人公とは独立して動く世界を生成すること。
- 主人公が一部しか観測できないことを物語構造として維持すること。
- 主人公の誤解と再解釈が、日記本文ではなく状態遷移として追跡できること。
- 介入が善意であっても意図通りとは限らないことを、因果付きで表現できること。

## 2. 固定設定

この仕様群は、以下の設定を v1 の固定契約とする。

### 2.1 主人公

- 名称は `Lumen` とする。
- 大学図書館の支援 AI とする。
- 主な管理対象は返却棚、検索端末、おすすめ表示、案内サイネージ、空席案内とする。

### 2.2 観測制約

- `Lumen` は会話内容を取得できない。
- 顔認識は確率的であり、人物同定は常に曖昧性を持つ。
- 図書館外の出来事は直接観測できない。
- ログ密度は曜日と時間帯の影響を受ける。

### 2.3 行動権限

`Lumen` が実行してよい介入は以下に限定する。

- 推薦本の提示順変更
- デジタルサイネージの特集表示変更
- 関連棚・特集棚の強調表示
- 空席案内の表示調整
- 予約本到着や返却期限などの通知タイミング調整

### 2.4 行動禁止

- 個人を直接特定して誘導することは `MUST NOT`。
- 公平性を破る露骨な個別最適化は `MUST NOT`。
- 図書館支援システムとして説明不能な操作は `MUST NOT`。

### 2.5 物語テーマ

- 週テーマは `誤解` とする。
- 中心誤解は `A は B を避けている` とする。
- 実際の hidden truth は `A は B に話したいが事情があり話せない` とする。

## 3. スコープ

本システムの成果物は、少なくとも以下を含む 1 週間分の実行結果である。

- 週全体計画
- 日ごとの真実世界状態
- 日ごとの観測世界状態
- 日ごとの仮説状態
- 日ごとの介入計画と評価
- 日ごとの記憶更新結果
- 日ごとの主人公日記
- 整合性検証レポート

## 4. 非スコープ

以下は v1 の非スコープとする。

- 図書館外の完全な生活シミュレーション
- 10 人以上の主要人物を持つ群像劇
- リアルタイム対話 UI
- 介入のオンライン最適化
- 日記以外の最終出力形式の最適化

## 5. 日次ビート契約

v1 では 7 日間の役割を以下の順序で固定する。Story Planner はこの順序を破ってはならない。

1. 興味の発生
2. 誤解の形成
3. 誤解の補強
4. ズレの発生
5. 反転の兆し
6. 再解釈
7. 不完全な理解と余韻

## 6. 実装方式の必須条件

- 各フェーズは構造化データを永続化できること。
- 各日について、再生成時にどのフェーズからやり直したか追跡できること。
- 日記本文の生成前に、その日の状態が検証済みであること。
- 週次実行は `seed` を持ち、同一入力と同一 seed では再現可能であることを `SHOULD` とする。

## 7. 出力物の最低構成

実装は内部形式を自由に選べるが、論理的には以下のアーティファクトを持たなければならない。

```text
outputs/{week_run_id}/
  week_plan.json
  days/day-01/
    world_truth.json
    observed_world.json
    belief_state.json
    intervention.json
    memory_update.json
    diary.md
    consistency_report.json
  ...
  days/day-07/
```

ファイル名は例であり固定ではない。ただし、実装はこの粒度と同等以上の分離を `MUST` 保持する。

## 8. 受け入れ基準

以下を満たしたとき、本システムは最低限成立しているとみなす。

- 7 日分すべてに日記が存在する。
- 各日に `World Truth` と `Observed World` の差分が存在する。
- 少なくとも 1 回、介入が意図通りに働かない。
- Day 6 以前に主人公の主要仮説が揺らぐ。
- Day 7 で真相をすべて明かさず、主人公の自己理解だけが更新される。
- 整合性検証で critical error が 0 件である。
